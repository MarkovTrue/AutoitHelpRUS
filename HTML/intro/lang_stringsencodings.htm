Строки и кодировки

Предисловие

Проблема внутреннего представления символов мучает компьютерную индустрию с тех пор, как IT стала повсеместной.
Изначально каждая компания использовала собственные соглашения и таблицы для представления текста и символов, делая совместимость кошмаром. Растущий спрос на поддержку большего числа символов, управляющих знаков и нелатинских письменностей только ухудшал ситуацию.
Кодировки и наборы символов напоминают игральные карты: таро и покер используют разный набор. После выбора набора нужно ещё создать рисунок (представление или кодировку) для каждой карты, чтобы каждый игрок мгновенно её узнавал.

Наборы символов

Сегодня все наборы символов делятся на 2 семейства: Unicode и кодовые страницы.

  • Unicode
    Также известен как ISO/IEC 10646. Этот огромный и сложный набор символов имеет неоспоримое преимущество универсальности: он стремится включить все символы и знаки, когда-либо использованные людьми, включая клинопись, иероглифы, клингонский, эмодзи и многое другое. Unicode версии 13.0 (03/2020) определяет 143 849 символов из 1 112 064 возможных кодовых точек, заданных стандартом.
    Unicode – больше, чем просто набор символов: он определяет способы работы с направленным текстом (слева направо или справа налево), сложным комбинированием глифов (графемы), процедуры сортировки (сравнения) строк, композицию/декомпозицию символов и многое другое.
  • Кодовые страницы
    Простая кодовая страница использует таблицу из N записей (обычно 256), сопоставляющую N символов или знаков числовому значению. В большинстве кодовых страниц каждый символ строки использует 8-битный байт – это удобно использовало доступную память, когда она была дефицитом. Однако ограничение в N символов мешало переносимости, и появилось множество несовместимых кодовых страниц. В итоге приходилось знать или угадывать кодировку полученного текстового файла, чтобы обработать его правильно.
    Чтобы преодолеть ограничение коротких кодовых страниц, делавших невозможным представление больших алфавитов (например, китайского), создали более обширные кодовые страницы с переменной шириной – кодировки MBCS (Multi-Byte Character Set).

    • Вопрос представления строк в памяти или файлах с заданным набором символов возник, когда IT начало использовать непростые кодовые страницы.

    Кодировки строк AutoIt

    Нативные строки AutoIt используют набор символов и кодировку UCS-2. Это подмножество Unicode, ограниченное BMP (Basic Multilingual Plane) – первыми 64K кодовых точек Unicode. Кодировка использует 16-битные единицы (каждый символ представляется значением unsigned short); кодовые точки в диапазоне U+D800..U+DFFF (суррогаты в UTF16) не имеют особого смысла и просто зарезервированы для частного использования.
    Обратите внимание: Windows работает с Unicode очень давно – Win 3.x, Win95, NT добавили DLL для UCS-2, XP и выше работают с UTF16-LE.

    Однако некоторым приложениям нужно работать со строками в других кодировках.

  • UTF8
    UTF8 – MBCS-представление (переменной длины) Unicode-строки последовательностью из 1–4 8-битных байтов. Это повсеместно используемая кодировка в веб-страницах, XML и т. п. – одновременно экономичная по объёму и способная закодировать весь набор символов Unicode.
  • Кодовые страницы
    В зависимости от языковых настроек Windows, внешних файлов или потоков, которые нужно обрабатывать или создавать, может потребоваться использовать строки в не-Unicode кодовой странице или преобразовывать их в неё/из неё.
  • OEM-кодовые страницы – традиционные "DOS"-кодовые страницы, по одной на каждое возможное языковое значение.
  • Windows-кодовые страницы – также называемые "ANSI"-кодовыми страницами; используются современной консолью Windows и определяются текущим языком, если не изменены командой DHCP. Заметьте, что консоль Windows также может принимать/отображать UTF8 после команды "DHCP 65001".
  • Специальные кодовые страницы – среди них MBCS, используемые во многих азиатских контекстах, и EBCDIC, используемый в большинстве мейнфреймов IBM.

    • Преобразование строк AutoIt из/в нативный UCS-2 в/из некоторой кодовой страницы.

    Для нужного преобразования можно использовать эти функции. Идентификатор кодовой страницы 65001 означает UTF8, но можно передать любой идентификатор, поддерживаемый Windows.
    Список кодовых страниц, поддерживаемых Windows, доступен здесь:

    https://docs.microsoft.com/en-us/windows/win32/intl/code-page-identifiers

    ; Для преобразования встроенной строки AutoIt (UCS-2) на некоторую кодовую страницу (по умолчанию UTF8):
    Func _StringToCodepage($sStr, $iCodepage = Default)
        If $iCodepage = Default Then $iCodepage = 65001
        Local $aResult = DllCall("kernel32.dll", "int", "WideCharToMultiByte", "uint", $iCodepage, "dword", 0, "wstr", $sStr, "int", StringLen($sStr), _
                "ptr", 0, "int", 0, "ptr", 0, "ptr", 0)
        Local $tCP = DllStructCreate("char[" & $aResult[0] & "]")
        $aResult = DllCall("Kernel32.dll", "int", "WideCharToMultiByte", "uint", $iCodepage, "dword", 0, "wstr", $sStr, "int", StringLen($sStr), _
                "struct*", $tCP, "int", $aResult[0], "ptr", 0, "ptr", 0)
        Return DllStructGetData($tCP, 1)
    EndFunc   ;==>_StringToCodepage

    ; Для преобразования строки из заданной кодовой страницы (по умолчанию UTF8) на встроенную строку AutoIt (UCS-2):
    Func _CodepageToString($sCP, $iCodepage = Default)
        If $iCodepage = Default Then $iCodepage = 65001
        Local $tText = DllStructCreate("byte[" & StringLen($sCP) & "]")
        DllStructSetData($tText, 1, $sCP)
        Local $aResult = DllCall("kernel32.dll", "int", "MultiByteToWideChar", "uint", $iCodepage, "dword", 0, "struct*", $tText, "int", StringLen($sCP), _
                "ptr", 0, "int", 0)
        Local $tWstr = DllStructCreate("wchar[" & $aResult[0] & "]")
        $aResult = DllCall("kernel32.dll", "int", "MultiByteToWideChar", "uint", $iCodepage, "dword", 0, "struct*", $tText, "int", StringLen($sCP), _
                "struct*", $tWstr, "int", $aResult[0])
        Return DllStructGetData($tWstr, 1)
    EndFunc   ;==>_CodepageToString


    Если нужно лишь преобразовывать нативные строки AutoIt в/из UTF8 (очень распространённый случай), можно использовать вот это:

    $sMyString = "Hello Χαίρετε こんにちは Привет xin chào हैलो مرحبا 你好 שלום வணக்கம்"
    $sUTF8String = BinaryToString(StringToBinary($sMyString & @LF, 4), 1)

    ; обратное преобразование:
    $sMyStringBack = BinaryToString(StringToBinary($sUTF8String & @LF, 1), 4)

    Советы

    Рекомендуется использовать кодировку UTF8 по умолчанию для исходных файлов: строки будут отображаться буквально и в исходном коде, и в Windows-контролах.
    Также рекомендуется выставить кодировку UTF8 в консоли SciTe4AutoIt3, если когда-либо потребуется отобразить символы или знаки, отсутствующие в кодовой странице Windows по умолчанию.

    Чтобы отправлять UTF8-строки в консоль SciTe, можно использовать такую функцию:

    ; Unicode-осведомленный ConsoleWrite для консоли SciTe UTF8
    Func _ConsoleWrite($s)
        ConsoleWrite(BinaryToString(StringToBinary($s & @LF, 4), 1))
    EndFunc   ;==>_ConsoleWrite


    Кроме того, если программа может использовать скомпилированный CUI-интерфейс *или* нескомпилированную консоль SciTe (например, для отладки), можно использовать вот это:

    ; Косвенная Unicode-осведомленная функция для UTF8 Scite или консоли CUI consolewrite
    Func __ConsoleWrite($s)
        Return (@Compiled ? _CUI_ConsoleWrite : _ConsoleWrite)($s)
    EndFunc   ;==>__ConsoleWrite

    ; Функция для консоли CUI consolewrite UTF16
    Func _CUI_ConsoleWrite(ByRef $s)
        Local Static $hCon = _CUI_ConsoleInit()
        DllCall("kernel32.dll", "bool", "WriteConsoleW", "handle", $hCon, "wstr", $s & @LF, "dword", StringLen($s) + 1, "dword*", 0, "ptr", 0)
        Return
    EndFunc   ;==>_CUI_ConsoleWrite

    ; Вспомогательная функция для консоли CUI consolewrite
    Func _CUI_ConsoleInit()
        DllCall("kernel32.dll", "bool", "AllocConsole")
        Return DllCall("kernel32.dll", "handle", "GetStdHandle", "int", -11)[0]
    EndFunc   ;==>_CUI_ConsoleInit

    Например, запустите этот пример с использованием функций выше – "Hello" должен корректно отображаться на разных языках одинаково и в MsgBox, и в консоли (SciTe или CUI):

    $sMyString = "Hello Χαίρετε こんにちは Привет xin chào हैलो مرحبا 你好 שלום வணக்கம்"
    __ConsoleWrite($sMyString)
    MsgBox(0, "", $sMyString)
